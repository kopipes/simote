Overview & Latar BelakangSimote adalah aplikasi web sederhana untuk menyimpan catatan pribadi, termasuk data sensitif seperti kredensial SSH dan pasangan login/password (misalnya akun server, akun aplikasi internal, dan lain-lain). Aplikasi ini ditujukan untuk individu atau tim kecil (multi-user) yang membutuhkan tempat penyimpanan terpusat yang aman, mudah dicari, dan cepat diakses, tanpa kompleksitas tools enterprise password manager.Masalah yang ingin diselesaikan: saat ini kredensial dan catatan teknis sering tersebar di banyak tempat (notes app, spreadsheet, chat pribadi, sticky notes) sehingga sulit dicari, rawan tercecer, dan tidak aman karena tidak terenkripsi. Simote menyatukan semuanya dalam satu tempat dengan kategori yang fleksibel, pencarian cepat, dan mekanisme reveal/copy yang aman serta nyaman.Value Proposition UtamaSatu tempat aman untuk semua catatan & kredensial Ñ tidak perlu lagi screenshot atau notes app biasa.Cari dan temukan kredensial dalam hitungan detik (search-first experience).Copy credential tanpa highlight manual Ñ satu klik langsung ke clipboard.Sensitive value disembunyikan secara default (asterisk) untuk mencegah shoulder-surfing.2. Tujuan Produk & Metrik Keberhasilan2.1 Tujuan (Goals)¥ Menyediakan tempat penyimpanan catatan & kredensial yang aman dengan autentikasi multi-user.¥ Memungkinkan pencarian catatan secara instan (real-time search) berdasarkan judul, isi, tag, atau kategori.¥ Mendukung kategori yang fleksibel Ñ user bisa membuat, mengedit, dan menghapus kategori sendiri.¥ Menyediakan UX copy-to-clipboard yang sangat cepat untuk credential (1 klik, tanpa seleksi manual teks).¥ Melindungi data sensitif secara visual (masking) dan secara teknis (enkripsi at-rest) di server/VPS.2.2 Success MetricsMetrikTargetCara UkurWaktu rata-rata user menemukan 1 note via search< 5 detikTime-to-result dari input search pertama sampai klik hasilWaktu copy credential (klik tombol copy s.d. tersalin)< 1 detik (instan)Response time aksi copy di frontendTingkat error login (gagal autentikasi tidak wajar)Dipantau, rate limit aktifLog autentikasiAdoption: rata-rata note dibuat per user per mingguBaseline awal, dievaluasi bulan 1Analytics jumlah note baru3. Target Pengguna¥ Developer / IT support yang sering menyimpan kredensial server, SSH key, akun database, akun staging/production.¥ Tim kecil (startup, agency) yang perlu berbagi referensi akun secara terpusat (multi-user, tiap orang punya akun login sendiri).¥ Individu yang ingin merapikan catatan pribadi (ide, todo, referensi) sekaligus kredensial akun pribadi dalam satu app.Catatan: pada versi awal (v1), setiap user memiliki akun dan data note miliknya bersifat privat per akun (tidak otomatis dibagikan ke user lain). Fitur sharing antar-user dapat dipertimbangkan di fase berikutnya Ñ lihat bagian Out of Scope.4. Scope4.1 In Scope (v1.0)¥ Web application, responsive (desktop-first, tetap nyaman dipakai di tablet/mobile browser).¥ Autentikasi multi-user: register, login, logout, dengan password ter-hash di database.¥ CRUD catatan (create, read, update, delete) dengan tipe konten: catatan teks biasa, kredensial SSH, dan pasangan login/password.¥ Manajemen kategori: tambah, edit nama, ubah warna, hapus kategori; assign note ke satu atau lebih kategori.¥ Pencarian instan (real-time, debounced) di judul, isi catatan, tag, dan nama kategori.¥ Masking otomatis untuk semua field bertipe sensitif (password, passphrase, private key) dengan toggle show/hide per field.¥ Copy-to-clipboard 1 klik per field, termasuk opsi "copy login saja", "copy password saja", atau "copy keduanya" (format siap-paste).¥ Status note: aktif, kadaluarsa/invalid (strikethrough), dan opsi hapus permanen.¥ Penyimpanan data di VPS menggunakan SQLite, dengan enkripsi pada kolom sensitif sebelum disimpan.4.2 Out of Scope (v1.0) Ñ Dipertimbangkan untuk Fase Berikutnya¥ Sharing/permission note antar-user (read-only share, team vault, role admin/member).¥ Aplikasi native mobile/desktop (saat ini hanya web app).¥ Sinkronisasi multi-server / multi-region, replikasi database.¥ Audit log detail (siapa membuka/copy credential apa, kapan) Ñ bisa jadi fitur keamanan fase 2.¥ Two-factor authentication (2FA) Ñ direkomendasikan untuk fase 2 mengingat sensitivitas data.¥ Import/export massal dari password manager lain (1Password, Bitwarden, KeePass, dll).5. User mendarat di Dashboard, melihat search bar besar di bagian atas (paling prioritas secara visual).7. User mengetik kata kunci (misal nama server, "ssh", atau nama aplikasi) ? hasil muncul real-time saat mengetik (tanpa perlu klik tombol search).8. Hasil pencarian ditampilkan sebagai list/card, dikelompokkan atau diberi label kategori serta tipe (Note / SSH / Login-Password).9. User klik salah satu card ? detail note terbuka (bisa berupa expand inline atau panel/modal).10. Field sensitif tampil sebagai ¥¥¥¥¥¥ dengan ikon mata di sampingnya; field punya ikon copy di kanan.11. User klik ikon copy ? value langsung tersalin ke clipboard, disertai micro-feedback (misal tooltip "Tersalin!" + ikon berubah jadi centang sesaat).5.3 Flow Menandai Credential Invalid12. Functional Requirements (Detail per Fitur)6.1 Autentikasi (Login & Password Awal)¥ FR-1: Sistem mendukung registrasi user baru dengan field minimal: username/email, password, konfirmasi password.¥ FR-2: Password user di-hash menggunakan algoritma standar industri (contoh: bcrypt/argon2) Ñ tidak pernah disimpan dalam bentuk plain text.¥ FR-3: Sistem mendukung login dengan session/token (contoh: JWT atau session cookie HttpOnly) dengan masa berlaku tertentu (misal 7 hari, dengan opsi "Ingat saya").¥ FR-4: Setelah 5 kali percobaan login gagal berturut-turut, sistem menerapkan rate limiting/cooldown sementara pada akun atau IP terkait.¥ FR-5: Tersedia fitur logout yang menghapus session aktif.¥ FR-6: (Direkomendasikan) Tersedia fitur "lupa password" minimal melalui reset manual oleh admin di v1, dengan rencana email-based reset di fase berikutnya.6.2 Manajemen Kategori¥ FR-7: User dapat membuat kategori baru langsung dari halaman manapun (misal tombol "+ Kategori" yang selalu terlihat di sidebar), tanpa perlu pindah ke halaman setting terpisah.¥ FR-8: Setiap kategori memiliki nama dan warna label (dipilih dari palet warna yang disediakan) untuk identifikasi visual cepat.¥ FR-9: User dapat mengedit nama/warna kategori dan menghapus kategori. Saat kategori dihapus, note yang terkait tidak ikut terhapus Ñ hanya kehilangan label kategori tersebut (atau dipindah ke kategori "Tanpa Kategori").¥ FR-10: Satu note dapat memiliki lebih dari satu kategori/tag sekaligus (many-to-many), untuk fleksibilitas pengelompokan.¥ FR-11: Sidebar menampilkan daftar kategori dengan jumlah note di masing-masing kategori, dan bisa diklik untuk filter cepat.6.3 Pencarian (Search)¥ FR-12: Search bar tersedia secara persisten di bagian atas halaman (sticky), dapat diakses dari halaman manapun di dalam aplikasi.¥ FR-13: Pencarian bersifat real-time/instant (debounce ~200Ð300ms) Ñ hasil muncul saat user mengetik, tanpa perlu menekan Enter.¥ FR-14: Pencarian mencakup: judul note, isi/catatan teks, nama field custom, nama kategori/tag, dan (untuk SSH) host/alamat server Ñ namun TIDAK mencakup isi field yang sensitif seperti password atau private key (agar value sensitif tidak ter-index dalam bentuk yang berisiko).¥ FR-15: Hasil pencarian menampilkan highlight kata kunci yang cocok serta badge tipe note (Catatan / SSH / Login-Password) dan kategori terkait.¥ FR-16: Mendukung filter tambahan: berdasarkan kategori, tipe note, dan status (aktif/invalid).6.4 Catatan Teks Biasa¥ FR-17: User dapat membuat catatan teks bebas dengan judul dan isi (rich text minimal: bold, italic, bullet list, link).¥ FR-18: Catatan teks biasa dapat diberi kategori/tag seperti tipe note lainnya.6.5 Kredensial SSH (Fitur Poin 1)¥ FR-19: User dapat menambah entri bertipe "SSH Credential" dengan field: Nama/Label (misal "Server Production - Data Model (Konsep Awal)Berikut struktur data konseptual untuk SQLite di VPS. Detail kolom dapat disesuaikan saat implementasi teknis.TabelKolom UtamaCatatanusersid, username/email, password_hash, created_atPassword wajib di-hash (bcrypt/argon2), tidak pernah disimpan plain textcategoriesid, user_id, name, color_hex, created_at1 user bisa punya banyak kategori; warna untuk identifikasi visualnotesid, user_id, type (note/ssh/login), title, status (active/invalid), created_at, updated_attype menentukan field tambahan apa yang relevan (lihat tabel detail)note_fieldsid, note_id, field_key, field_value_encrypted, is_sensitive (bool)Pendekatan generic field agar fleksibel menambah field baru tanpa migrasi schema; value sensitif dienkripsi sebelum disimpannote_categoriesnote_id, category_idTabel pivot many-to-many antara note dan kategoriCatatan Keamanan Data PentingKarena data disimpan di VPS dengan SQLite, kolom-kolom sensitif (password, private key, passphrase) WAJIB dienkripsi (contoh: AES-256) sebelum ditulis ke database, bukan disimpan sebagai plain text Ñ file SQLite yang ter-leak/diakses langsung tetap tidak membocorkan credential.Kunci enkripsi (encryption key) disimpan terpisah dari file database, idealnya melalui environment variable atau secret manager di VPS, bukan hard-coded di kode aplikasi.Koneksi aplikasi wajib menggunakan HTTPS (TLS) Ñ tidak ada pengiriman password/credential melalui koneksi HTTP biasa.Lakukan backup database terjadwal (terenkripsi) untuk mencegah kehilangan data akibat masalah VPS.8. Non-Functional Requirements8.1 Keamanan¥ Enkripsi at-rest untuk semua field sensitif (lihat bagian 7).¥ Enkripsi in-transit (HTTPS/TLS wajib, tidak ada fallback ke HTTP).¥ Session/token memiliki masa berlaku (expiry) dan dapat di-revoke (logout memutus akses token).¥ Tidak ada data sensitif yang tercatat di server-side log (misal request log tidak boleh mencatat isi password).¥ Rate limiting pada endpoint login/register untuk mencegah brute force.8.2 Performa¥ Hasil pencarian harus muncul dalam < 300ms untuk dataset hingga beberapa ribu note per user (mengingat SQLite, perlu indexing yang baik pada kolom judul/isi/kategori).¥ Initial page load (setelah login) target < 2 detik pada koneksi broadband standar.8.3 Usability¥ Tidak ada fitur utama yang membutuhkan lebih dari 2 klik dari Dashboard (misal: tambah note baru, tambah kategori, copy credential).¥ Semua aksi destruktif (hapus permanen) wajib ada konfirmasi; aksi reversible (tandai invalid) tidak perlu konfirmasi berat agar tetap cepat dipakai.8.4 Skalabilitas (Catatan untuk Fase Berikutnya)SQLite cocok untuk fase awal dengan jumlah user kecil-menengah dan akses tidak terlalu konkuren. Jika jumlah user/tim bertambah signifikan atau dibutuhkan akses concurrent yang tinggi, pertimbangkan migrasi ke PostgreSQL/MySQL di fase berikutnya tanpa mengubah struktur data inti secara drastis.9.
